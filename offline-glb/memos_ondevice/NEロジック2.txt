NEゲートウェイ 内部ロジック仕様（Cursor用・完全版）

0. 大原則（破ると全体崩壊）
	1.	NEはURLを持たない（HTTPの顔をしない）
	2.	アプリはNEを知らない（参照・import・文字列露出ゼロ）
	3.	外部説明は tomoriRun のみ（翻訳HTTPエンドポイント）
	4.	**NEの仕事は“判断と制御”**であり、“翻訳”そのものではない
	5.	失敗は静かに（暴走・無限リトライ禁止）

⸻

1) 全体アーキテクチャ（役割固定）
	•	Client (App)
	•	送る：入力テキスト + 言語指定（任意）
	•	受け取る：{ translatedText: string }
	•	ロジック：UI/状態管理のみ
	•	禁止：NE語彙、ゲートウェイ、最適化用語
	•	Public Endpoint (tomoriRun)
	•	“口”＝HTTP窓口
	•	返す：translatedText
	•	中身は自由に差し替え可（ただし外から見える契約は固定）
	•	NE (Internal Control Plane / Offline)
	•	“思考層”＝判断・ルーティング・安全制御
	•	直接クライアントから呼ばれない
	•	URLを持たない
	•	オフラインで成立する設計を最優先

⸻

2) NEの目的（何を最適化するか）

NEが最適化する対象は「翻訳精度」だけじゃない。優先順位はこれ：
	1.	安全性（暴走防止 / 無限リトライ防止 / 予算保護）
	2.	一貫性（同入力→同出力、再現性）
	3.	可用性（落ちない、固まらない）
	4.	品質（自然さ / 意味保持）
	5.	速度（体感、ただし安全性の後）

⸻

3) “NEゲートウェイ”の内部ステートマシン（最重要）

NEは必ずこの順で状態遷移する（省略禁止）：

State: IDLE
	•	待機。入力を受ける。

State: PREFLIGHT
	•	入力検証（空/長すぎ/禁止文字）
	•	ネット状態判定（online/offline）
	•	予算・回数制限判定（例：日次回数、課金状態）
	•	端末状態（低電力/バックグラウンド/タイムアウト枠）

State: ROUTE_SELECT
	•	ルート決定（後述）
	•	ルートは必ず1つに確定（ブレさせない）

State: EXECUTE
	•	実行は 単発（リトライはNEが明示的に許可した場合のみ）
	•	実行結果を正規化（整形/空判定）

State: POSTCHECK
	•	“成功条件”評価
	•	失敗ならフォールバック（最大1回）

State: COMMIT
	•	キャッシュ保存（必要なら）
	•	ログ（内部のみ、外部露出ゼロ）

State: RETURN
	•	{ translatedText } だけ返す（余計なメタ禁止）

⸻

4) ルーティング規則（ROUTE_SELECTの決め方）

NEは以下の優先順でルートを選ぶ。複数併用しない。

Route A: OFFLINE_DICT（最優先）
	•	条件：辞書/フレーズテーブルで処理可能
	•	出力：即返す
	•	目的：コスト0、落ちない

Route B: OFFLINE_ENGINE（将来枠）
	•	条件：ローカル推論/ローカル変換で成立
	•	出力：ローカル生成

Route C: ONLINE_API（tomoriRun経由）
	•	条件：online、かつA/B不可
	•	注意：NEはURLを持たないが、外側の窓口（tomoriRun）はある
	•	NEは「オンラインを使う許可」を出すだけ（実際のHTTPは窓口側が行う）

Route D: FAILSAFE_TEXT
	•	条件：全部ダメ
	•	出力：固定の安全文（例：「通信できません。ネット環境を確認してください。」）
	•	無限試行しない

⸻

5) “エラー祭り”を永遠に起こさないための規律（超重要）

Cursorが一番守るべきのはこれ。

5.1 単発実行の原則
	•	1タップ = 1リクエスト
	•	進行中は二度押し無効（UI側ガード）

5.2 冪等性（requestId）
	•	リクエストには requestId を付ける（UUID or monotonic int）
	•	activeRequestId を保持し、古いレスポンスは捨てる

5.3 リトライ禁止（例外はNEが明示）
	•	タイムアウト・500でも勝手に再実行しない
	•	例外：NEが「1回だけフォールバック」許可する場合のみ

5.4 出力の上書き規則
	•	translatedText は常に上書き（+=禁止）
	•	実行開始時に前回結果を消す（残像禁止）

⸻

6) データ契約（外部に見えるのはこれだけ）

Request（最小）

{
  "text": "こんにちは",
  "source": "ja",
  "target": "en"
}

Response（固定）

{
  "translatedText": "Hello"
}

これ以外のフィールド追加は慎重に（審査・互換性の敵）
	•	NEやルート情報は絶対に返さない

⸻

7) 実装の分離ルール（Cursor作業規約）

7.1 クライアント（Flutter）
	•	触っていい：API呼び出し関数、状態管理
	•	禁止：NE語彙・gateway_core・NE_Gatewayクラスの復活
	•	URL：tomoriRun 1本のみ

7.2 サーバー窓口（tomoriRun）
	•	役割：HTTP受付、翻訳結果を返す
	•	禁止：NE関連ファイル同梱、NE語彙露出
	•	実装は最小でいい（審査最優先）

7.3 NE（内部）
	•	ここは別管理（ローカル）
	•	外部に混ぜない

⸻

NE露出ゼロ監査
	•	文字列・import・ファイル存在を全チェック

キーワード（最低限）：
NE, NEXTBASE, Gateway, NE_Gateway, optimizer, SOL, Stella, layer
